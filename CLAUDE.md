## What this project is

**Farm Data Hub** — a Next.js 13 (App Router) demo that integrates with the John Deere Operations Center API. Users authenticate with Supabase, connect their John Deere account via OAuth 2.0, choose an organization, then browse their fields, boundaries, and operations (harvest, seeding) on a map-first interface.

The app is deployed to **Vercel**.

### Data Flow

User authenticates via Supabase Auth → `auth-context` provides session → components call `lib/john-deere-client.ts` with Bearer JWT → Edge Functions validate JWT via `getAuthenticatedUser()` → Edge Functions call John Deere API using stored OAuth tokens (auto-refresh in `_shared/john-deere.ts` `getValidToken()`) → data persisted to PostgreSQL (`operations_center` schema, NOT `public`) → components read stored data → Mapbox GL renders pre-converted GeoJSON field boundaries from DB without round-tripping through JD.

### External API Calls

- `GET/POST https://api.deere.com/platform/...` — John Deere Operations Center API, triggered on import action via Edge Functions
- `POST https://signin.johndeere.com/oauth2/.../v1/token` — OAuth token exchange/refresh, called from Edge Functions
- `POST https://nuxofsjzrgdauzriraze.supabase.co/functions/v1/*` — Supabase Edge Functions, called from `lib/john-deere-client.ts` on user interaction
- `GET https://api.mapbox.com/...` — Mapbox tiles, loaded on map page render
- `GET <pre-signed S3 URL>` — JD shapefile download via 307 redirect, triggered by `john-deere-irrigation` shapefile-status polling

## Guardrails

### Universal (apply to all projects)

1. Never hardcode API keys or secrets — use `.env` + `.env.example` from first commit
2. Every API route needs CORS origin validation, input validation, and auth
3. Write RLS/Firestore rules the same day you create a table
4. Never use `innerHTML` with dynamic data — use `textContent` or framework escaping
5. API proxies must restrict destination URLs with an allowlist (SSRF prevention)
6. Auth cookies must contain signed claims (HMAC), not plain booleans
7. If a secret was ever committed to a public repo, rotate it immediately
8. No `console.log` with sensitive data in production
9. Run `npx prettier --write .` before committing — keep a `.prettierrc` in the repo root

### Project-Specific

- **Public repo:** This repo is PUBLIC on GitHub. Never commit `.env*` files; rotate immediately if you do — already gitignored, verified clean in history.
- **CORS:** Restrict `Access-Control-Allow-Origin` in `_shared/cors.ts` to `https://operations-center-api-demo.vercel.app` (and `http://localhost:3000` for dev) — wildcard is verified live in production.
- **Errors:** Edge function catch blocks must return generic messages only — log full errors server-side, never forward `error.message` or `error.stack` to HTTP responses (currently happens in all 4 functions; `john-deere-import/index.ts:687` also leaks the stack).
- **Rate limiting:** Add rate limiting to any new edge function endpoints (especially anything that triggers paid JD API calls).
- **Input validation:** All edge function request bodies must be validated with Zod before use.
- **Server-side auth:** Add `middleware.ts` at the project root using `@supabase/ssr` to protect authenticated routes server-side.
- **OAuth scopes:** Drop `ag2 ag3 org2 work2` from `getJohnDeereAuthUrl` until write is needed — broader scopes than current features require.
- **Do not add to `supabase/functions/john-deere-import/index.ts`** (689 lines, still growing) — split into per-action modules before adding features.
- **`components/dashboard/irrigation-analysis.tsx`** is now 660 lines and `app/(app)/progress/page.tsx` is 639 lines — both warrant splitting before next round of features.
- **Next.js residual CVEs:** 5 vulns remain in next 13.5.x (postcss + 4 high in next itself). Fix path is `npm audit fix --force` → next 16.2.5 (breaking major). Plan the bump as a deliberate migration, not a one-off audit fix.
